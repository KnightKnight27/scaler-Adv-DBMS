## Objective
The goal of this lab is to inspect how SQLite stores table and index data on disk. The submission creates a small campus database, fixes the page size to 1024 bytes, and records the page layout using SQLite metadata and a hex dump.

## Files
- `campus.sql`: SQL script used to create and populate the database.
- `campus.db`: SQLite database generated from the script.
- `campus.hex`: Hex dump of the generated database file.
- `README.md`: Notes and observations from the inspection.

## Recreate The Database

```powershell
sqlite3 lab4/campus.db ".read lab4/campus.sql"
```

The script creates three tables:

| Table | Purpose |
| --- | --- |
| `courses` | Course catalog with course code, title, and credits |
| `students` | Student records with roll number, branch, semester, and CGPA |
| `enrollments` | Mapping between students and courses |

It also creates indexes on `students.roll_no` and `enrollments.student_id`.

## Basic Database Metadata

Commands used:

```sql
PRAGMA page_size;
PRAGMA page_count;
SELECT name, type, rootpage
FROM sqlite_master
WHERE type IN ('table', 'index')
ORDER BY rootpage;
```

Observed output:

```text
page_size  = 1024 bytes
page_count = 25 pages
file_size  = 25600 bytes
```

The file size matches the page calculation:

```text
1024 bytes x 25 pages = 25600 bytes
```

## Page Type Inspection

The `dbstat` virtual table was used to inspect page ownership and page type:

```sql
SELECT pageno, name, pagetype, ncell, payload, unused, mx_payload
FROM dbstat
ORDER BY pageno;
```

Key observations:

| Page | Object | Page Type | Cells | Notes |
| --- | --- | --- | --- | --- |
| 1 | `sqlite_schema` | internal | 1 | Database header and schema B-tree root |
| 2 | `courses` | leaf | 5 | Stores all course rows |
| 4 | `students` | internal | 5 | Root page for student table B-tree |
| 6 | `enrollments` | internal | 1 | Root page for enrollment table B-tree |
| 9 | `idx_enrollments_student` | leaf | 120 | Compact index entries |
| 11-13 | `students` | leaf | 22 each | Student table row storage |
| 18-20 | `students` | leaf | 22, 22, 10 | Remaining student rows |
| 23-24 | `enrollments` | leaf | 70, 50 | Enrollment rows split across two leaf pages |
| 25 | `sqlite_stat1` | leaf | 4 | Statistics generated by `ANALYZE` |

## Header And B-tree Notes

The first bytes of the database begin with the SQLite file signature:

```text
53 51 4C 69 74 65 20 66 6F 72 6D 61 74 20 33 00
```

This decodes to:

```text
SQLite format 3
```

The next important header bytes show a 1024-byte page size:

```text
00 04
```

For B-tree pages, the first byte identifies the page kind:

| First Byte | Meaning |
| --- | --- |
| `0x05` | Interior table B-tree page |
| `0x0D` | Leaf table B-tree page |

Examples from this database:

| Page | First Bytes | Interpretation |
| --- | --- | --- |
| Page 2 | `0D 00 00 00 05 ...` | Leaf table page with 5 cells |
| Page 4 | `05 00 00 00 05 ...` | Interior table page with 5 cells |
| Page 11 | `0D 00 00 00 16 ...` | Leaf table page with 22 cells |

## What I Learned

- SQLite stores data in fixed-size pages; here each page is 1024 bytes.
- A table grows from a single leaf page into an interior B-tree root plus multiple leaf pages.
- Index pages are separate B-trees and can be more compact than table pages because they store key references instead of full rows.
- The `sqlite_master` table shows logical objects, while `dbstat` reveals the physical page-level layout.
- Hex dumps confirm that the database file starts with the SQLite signature and then stores B-tree page headers and cells page by page.
