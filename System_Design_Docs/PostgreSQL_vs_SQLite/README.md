## 1. Problem Background

Database management systems are designed around their operational environment and target workloads. The architectural split between **PostgreSQL** and **SQLite** represents a fundamental division in database system design: the client-server multi-user model versus the embedded single-user library model.

### PostgreSQL: Enterprise Relational Power
PostgreSQL was conceived in 1986 at UC Berkeley by Michael Stonebraker as the successor to the INGRES project. The primary goal was to address the limitations of existing relational database models, particularly their inability to handle complex user-defined data types, object-oriented concepts, and extensible access methods. It was designed from the ground up to operate as an enterprise-grade, multi-user, networked database system where data integrity, complex querying capabilities, and concurrent transaction execution are paramount.

### SQLite: Zero-Configuration Embedded Engine
SQLite was created in 2000 by D. Richard Hipp while working on a US Navy guided-missile destroyer. The objective was to build a database system that did not require administrative setup, installation, or a running daemon. It needed to be directly embedded within application processes, operate reliably across system crashes, and read/write directly to standard disk files. It solved the problem of replacing direct file-based serialization (like custom binary formats or CSVs) with a structured, transactional SQL engine.

## 2. Architecture Overview

### High-Level Architectural Flow

```mermaid
graph TD
    subgraph PostgreSQL (Client-Server Model)
        Client[Client Application] <-->|TCP/IP Socket / Unix Domain| Postmaster[Postmaster Daemon / Dispatcher]
        Postmaster -->|Spawns| Backend[Backend Worker Process]
        Backend <--> SharedMem[Shared Memory Pool <br> - Buffer Cache <br> - Lock Table <br> - WAL Buffer]
        SharedMem <--> DiskFiles[Postgres Data Directories <br> - Heap Files <br> - Indexes <br> - WAL Logs]
    end

    subgraph SQLite (Embedded Model)
        App[Client Process Space]
        subgraph SQLite Library (libsqlite3)
            App --> API[SQL Compiler Frontend <br> - Tokenizer / Parser <br> - Code Generator]
            API --> VDBE[Virtual Database Engine <br> - Bytecode Interpreter]
            VDBE --> BTree[B-Tree Layer]
            BTree --> Pager[Pager Cache Layer]
            Pager --> VFS[Virtual File System Layer]
        end
        VFS <--> SingleFile[Single Database File <br> .sqlite / .db]
    end
```

### Main System Components

#### PostgreSQL Components:
1. **Postmaster Daemon (Control Process):** Listens on a designated port (default `5432`), authenticates clients, and forks a dedicated backend worker process (`postgres`) for each active connection.
2. **Backend Worker Processes:** Independent OS processes execution paths representing individual clients. They execute SQL statements, manage local transaction states, and coordinate memory access.
3. **Shared Memory Pool:** A shared memory segment utilized by all backend processes. It holds the shared buffer cache, global lock tables, transaction status cache (CLOG), and Write-Ahead Log (WAL) buffers.
4. **Background Processes:** System helper processes including the **Checkpointer** (flushes dirty pages), **Background Writer** (periodically writes pages to minimize checkpoint load), **WAL Writer** (flushes WAL buffers to disk), and **Autovacuum Launcher** (spawns workers to clean dead tuples).

#### SQLite Components:
1. **Tokenizer & Parser:** Converts SQL statements into an Abstract Syntax Tree (AST).
2. **Code Generator / Optimizer:** Evaluates index usage and query paths, generating optimized bytecode representation.
3. **Virtual Database Engine (VDBE):** A virtual machine that executes the generated bytecode. Almost all operations affecting B-Trees run as instructions inside the VDBE.
4. **B-Tree & Pager Layer:** The B-Tree module organizes the data on disk pages, while the Pager manages the in-memory cache, read/write locks, and rollback/WAL logging.
5. **Virtual File System (VFS):** An abstraction layer that translates SQLite's page read/write requests into operating-system-specific system calls (POSIX, Win32), making SQLite highly portable.

## 3. Internal Design

### Database File Organization & Page Layout

| Architectural Feature | PostgreSQL | SQLite |
| :--- | :--- | :--- |
| **Page Size** | 8 KB (Default, compile-time configurable) | 4 KB (Default, runtime configurable from 512B to 64KB) |
| **Storage Organization** | Heap-based (tuples placed in any page with free space) | Index-organized B-Tree (Rowid table) or Clustered B+Tree (`WITHOUT ROWID`) |
| **Header Allocation** | Top of the page (PageHeaderData) | Top of the page (varies by page type, first page has 100-byte DB header) |
| **Tuple Growth Direction**| Grows from the bottom of the page upwards | Cells grow from the bottom of the page upwards |
| **Pointer Array** | Line Pointers (linp) at the top grow downwards | Cell Pointers at the top grow downwards |

#### PostgreSQL Page Layout
A PostgreSQL page contains a header (`PageHeaderData`) containing LSN metadata, free space offsets, and flags. Right after the header is the line pointer array (pointers to tuples). The actual tuples are written starting from the end of the page and grow towards the top. The area between the line pointers and the newest tuple is the **Free Space**.

#### SQLite Page Layout
SQLite database files consist of one or more pages. Page 1 contains a 100-byte database header. B-Tree pages are structured with a page header (specifying page type, cell count, freeblock pointers, and cell content area offset), followed by the cell pointer array (which grows downwards). Cells (which contain key-value pairs representing records) are allocated from the bottom of the page upwards.

### Table Storage & Index Organization

#### PostgreSQL: Non-Clustered Heap Tables
* **Heap Storage:** PostgreSQL tables are stored as "heaps." Tuples are placed into pages based solely on space availability.
* **Tuple Identification:** Every row is uniquely identified by a physical address called a **TID (Tuple ID)**, consisting of a tuple's block number and its offset index within that block (e.g., `(0, 1)`).
* **Index Reference:** All indexes (B-Tree, GiST, GIN, etc.) store the indexed key along with the corresponding TID. A primary key lookup requires two steps: traversing the B-Tree index to retrieve the TID, and then accessing the physical heap page.

```
PostgreSQL Index Lookup:
[Index B-Tree Search] ---> Returns TID (Block #, Offset) ---> Direct Read of Heap Page
```

#### SQLite: B-Tree Clustered Storage
* **Rowid Tables:** By default, SQLite tables are structured as B-Tree structures keyed by a hidden 64-bit signed integer called the `rowid`. If a table declares an `INTEGER PRIMARY KEY`, it becomes an alias for the `rowid`.
* **WITHOUT ROWID Tables:** Users can specify `WITHOUT ROWID` to create a clustered B+Tree. In this layout, the primary key dictates the B-Tree index structure, and the actual column data is stored directly in the leaf pages of this B+Tree.
* **Secondary Index Cost:** In a `WITHOUT ROWID` table, secondary indexes store the primary key values, meaning a secondary index search requires a secondary lookup in the primary B-Tree. In standard `rowid` tables, secondary indexes map keys directly to the `rowid`.

### Transaction Management & Concurrency Control

#### Concurrency Models

* **PostgreSQL (MVCC):** 
  PostgreSQL achieves concurrency via Multi-Version Concurrency Control. Writers do not block readers, and readers do not block writers. Every transaction reads from a snapshot of the database taken at a specific point in time. Physical rows (tuples) are never updated in-place; instead, a new version of the row is inserted, marked with the current transaction ID (`xmin`), and the old row is marked as expired (`xmax`).
* **SQLite (Coarse-Grained Locking & WAL):**
  SQLite's concurrency is governed by lock states applied to the entire database file.
  * **Rollback Journal Mode:** Concurrency is highly restrictive. Multiple processes can read concurrently, but only one process can write, and writers block readers completely during commits.
  * **Write-Ahead Log (WAL) Mode:** SQLite splits the database access. Writes are written to a separate `.db-wal` file. Readers can read from the main database file while a writer appends to the WAL file, enabling concurrent reads and writes. However, only a single write transaction can execute at any time.

#### SQLite Rollback Journal Lock Transitions
During a transaction in rollback mode, the pager transitions the database file through five lock states:

```
UNLOCKED ---> SHARED (Read Only) ---> RESERVED (Intention to Write) ---> PENDING (Drain Readers) ---> EXCLUSIVE (Commit)
```

1. **SHARED:** Allows multiple connections to read data. No modifications are allowed.
2. **RESERVED:** Acquired by a connection intending to write. Only one reserved lock can exist at a time, but existing SHARED readers can continue reading.
3. **PENDING:** The writer is ready to commit. It waits for all active SHARED locks to release. No new SHARED locks are allowed.
4. **EXCLUSIVE:** The writer has sole access to write changes back to the main database file.

## 4. Design Trade-Offs

### 1. Process Isolation vs. Shared Address Space
* **PostgreSQL (Process-Per-Connection):**
  * *Advantage:* High isolation. If a backend worker crashes due to a bug or memory corruption, it only terminates that connection. The Postmaster recovers and other clients continue unaffected.
  * *Disadvantage:* High resource overhead. Creating a process requires duplicating file descriptors, memory mappings, and thread descriptors. Max connection limits are restricted (typically 100-500) without external pooling systems like PgBouncer.
* **SQLite (Library-in-Process):**
  * *Advantage:* Zero process management overhead. SQL execution occurs inside the client thread, meaning no context switching, IPC, or TCP network round-trips.
  * *Disadvantage:* Vulnerability. A crash in the application process (e.g., a segmentation fault) during a write operation could leave lock files orphaned or require rollbacks upon restart. Memory leaks or buffer overflows in the application can corrupt database memory structures.

### 2. Fine-Grained (Row-Level) vs. Coarse-Grained (File-Level) Locking
* **PostgreSQL:**
  * Uses a highly complex lock manager that tracks locks on relations, pages, and individual tuples using a shared-memory hash table. This allows massive parallel execution across hundreds of cores.
* **SQLite:**
  * Uses database-level locks. While light on memory and CPU overhead, it introduces a strict concurrency bottleneck. SQLite is not suited for write-heavy multi-user systems.

### 3. Storage Reclamation: Vacuuming vs. Auto-Vacuuming/Compaction
* **PostgreSQL:**
  * Because updates create new tuple versions in the heap, PostgreSQL requires a `VACUUM` process to mark dead tuples as free space. If `VACUUM` cannot keep up, tables suffer from severe "bloat," degrading scan times.
* **SQLite:**
  * In rollback mode, SQLite modifies pages in-place. If rows are deleted, they are added to a free list inside the file. Over time, the database file size does not decrease unless the `VACUUM` command is executed manually or `auto_vacuum` is enabled, which reorganizes pages but increases write overhead.

## 5. Observations & Experiments

### Transaction Batching Performance in SQLite

One of the most dramatic architectural demonstrations in SQLite involves transaction boundary management. Because SQLite defaults to auto-commit mode (wrapping each individual SQL statement in an implicit transaction), running consecutive insert operations without explicit transactions forces SQLite to sync the rollback journal to disk for every single write.

#### Scenario: Inserting 10,000 Records
We compare the throughput of inserting 10,000 rows into a simple table:
`CREATE TABLE test_table (id INTEGER PRIMARY KEY, val TEXT);`

1. **Approach A (Implicit Transactions / Auto-Commit):**
   ```sql
   INSERT INTO test_table VALUES (1, 'val1');
   INSERT INTO test_table VALUES (2, 'val2');
   -- ... repeated 10,000 times
   ```
2. **Approach B (Explicit Transaction Batching):**
   ```sql
   BEGIN TRANSACTION;
   INSERT INTO test_table VALUES (1, 'val1');
   INSERT INTO test_table VALUES (2, 'val2');
   -- ... repeated 10,000 times
   COMMIT;
   ```

#### Observed Performance Profile

```
Throughput (Inserts per Second)
Approach A: ██ 20 - 50 inserts/sec
Approach B: ██████████████████████████████ 45,000+ inserts/sec
```

#### Technical Analysis
* In **Approach A**, SQLite must create a journal file, write the original page state, perform the insert, flush the change to the journal, copy the modified page to the main database file, flush the database file, and then delete the journal for *each* record. This is bounded by the physical rotational/flash speed of the disk's `fsync` execution limits (typically ~50-100 writes/sec on traditional magnetic disks, slightly higher on SSDs depending on cache).
* In **Approach B**, SQLite performs all modifications in-memory inside the Pager's cache. Only a single write sequence, journal write, and `fsync` flush occur during the final `COMMIT` operation. This reduces the IO bottleneck, shifting performance from disk-bounded to memory-bounded.

## 6. Key Learnings

1. **Architecture Dictates the Use Case:** PostgreSQL is designed for network-based, highly concurrent operations where data safety across multiple clients is handled by process separation and row locks. SQLite is optimized to be a fast, portable SQL engine embedded within a single program interface.
2. **The Cost of Concurrency:** Providing fine-grained concurrency (like PostgreSQL's MVCC and Row Locks) requires significant metadata overhead (tuple headers, lock managers, shared memory coordination). Conversely, SQLite keeps metadata minimal at the cost of parallel write scaling.
3. **Clustered Storage Trade-Off:** SQLite's index-organized table layout makes primary key access fast but adds overhead to secondary index lookups because they must map to the key or rowid. PostgreSQL's heap organization decouples physical storage from indexing, which simplifies updates but introduces additional IO hops for primary index lookups.
